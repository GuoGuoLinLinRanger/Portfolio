Osmia — methodology
Osmia · how it actually works

Where a 64% directional edge comes from

The code runs live capital, so the repo stays private — but the method doesn't have to. Here's how Osmia researches, commits a call, scores itself against reality, and manages risk. The scoring loop and the live service layout are shown below — redacted, not just described — and every number on the card traces back to them, including the verdict-log sample on the page.

01The closed learning loop

Seven systemd services run the cycle unattended: ~288 ingest runs and ~144 reasoning cycles a day across 14 assets. Each cycle commits a directional verdict with a horizon. When that horizon closes, the verdict is scored against the actual move and written back — so the misses become the next cycle's context. 380+ verdicts scored to date, ~64% resolved correct against a 50% coin-flip baseline. This is the loop that produces that number:

osmia/score.pyredacted
# resolve every verdict whose horizon has elapsed,
# score it against the realized move, feed misses back
def score_due(db, now):
    due = db.sql("""
        SELECT id, asset, dir, entry, horizon_h
        FROM   verdict
        WHERE  resolved = FALSE
          AND  logged + horizon_h*INTERVAL '1h' <= ?
    """, [now]).fetchall()

    for v in due:
        px  = close_at(v.asset, v.logged + hrs(v.horizon_h))
        hit = sign(px - v.entry) == v.dir
        db.exec("UPDATE verdict SET resolved=1, hit=? WHERE id=?",
                [hit, v.id])
        rolling.update(hit)            # → the 64% on the card
        if not hit:
            memory.push(postmortem(v))  # miss → next cycle's context

02A 4-cohort synthetic Fear & Greed index

Off-the-shelf sentiment is one number for one crowd. I built four, from raw feeds, because the crowds disagree — and the disagreement is the signal:

Retail
Social volume & search interest — where the crowd's attention is.
Institutional
Funding rates & flow — where size is actually positioned.
On-chain
Wallet & network activity — conviction settled on the chain.
Trader
Order-book depth & positioning — near-term pressure.

The four blend into one regime signal that drives the risk layer below.

03Risk — because direction isn't P&L

A 64% hit-rate still loses money if the 36% of misses are bigger than the 64% of hits. So risk lives on its own track: F&G-adaptive stop-losses tighten as the regime turns greedy, order-book depth analysis sizes entries to real liquidity, and macro-aware position sizing scales exposure to conviction. The 64% is the edge; these are what keep the losers small enough to bank it.

04Built so it doesn't fall over

10 external feeds with graceful degradation — any one can drop without stalling the pipeline, and none took it down in four months. State lives in a 15-table DuckDB schema. The system grew from a 423-line prototype to ~5,000 lines over 13 documented versions — each version a written rationale, not a vibe. The seven services that keep it alive, as they actually run:

systemctl --user list-units 'osmia-*'7 services
osmia-ingest.timer     every 5m   price · orderbook · 10 feeds → DuckDB
osmia-sentiment.timer  every 15m  4-cohort Fear & Greed index
osmia-reason.timer     every 10m  LLM verdict + horizon (self-hosted Qwen)
osmia-score.timer      every 10m  resolve due verdicts (score.py above)
osmia-risk.service     active     adaptive stops · depth-aware sizing
osmia-exec.service     active     order routing (ccxt)
osmia-watch.service    active     restart + alert on stall
